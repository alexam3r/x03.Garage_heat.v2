# Garage Heat v2

Прошивка ESP8266 (Wemos D1 mini) для поддержания температуры в гараже/месте
зимнего хранения мотоцикла: тепловая пушка с адаптивным duty-циклом нагрева,
вспомогательный нагреватель, многоуровневая защита радиатора SSR от перегрева
и управление/телеметрия по MQTT.

Полное техническое задание — в [`CLAUDE.md`](CLAUDE.md). Пошаговый план
реализации и статус каждой задачи — в [`progress.md`](progress.md).

## Статус

Реализация завершена (17/17 задач плана), 37/37 unit-тестов проходят,
прошивка собирается под esp8266. Проект готов к bring-up на реальном железе.

## Возможности

- Автомат тепловой пушки: гистерезис ±0.5°C по целевому датчику, duty-цикл
  нагревательного элемента (ON/OFF), адаптирующийся по датчику обдува, и
  time-based cooldown вентилятора после выключения элемента.
- Вспомогательный нагреватель со своим термоконтуром по уличному воздуху,
  включаемый/выключаемый по MQTT-флагу и блокируемый при аварии радиатора.
- Многоуровневая защита радиатора SSR: включение обдувочного вентилятора по
  порогу; безусловное аварийное отключение всех нагрузок (элемент пушки,
  вспом. нагреватель, вентилятор пушки) — как по критическому порогу, так и
  по детекции отказа вентилятора охлаждения (промежуточный порог +
  минимальное время работы, без тахометра/датчика тока, срабатывает раньше
  критического); автоматический сброс аварии по остыванию.
- Пуск нагрева пушки дополнительно блокируется, пока активна любая авария
  радиатора (не только критическая) — иначе автомат нагрева и радиаторная
  защита «спорили» бы за состояние вентилятора пушки на каждой итерации.
- Жёсткий программный интерлок: нагревательный элемент пушки физически не
  может включиться в коде, если выход вентилятора пушки выключен.
- Watchdog связи: перезагрузка устройства, если WiFi+MQTT не восстанавливаются
  дольше 5 минут, с безопасным отключением всех силовых выходов перед
  перезапуском.
- Телеметрия в MQTT — по топику на каждую величину (`garage/heat/<имя>/state`,
  без JSON): все температуры, состояния выходов, таргеты/гистерезисы, флаги
  аварий, uptime, heap, RSSI, состояние WiFi/MQTT. Публикация раз в 60 секунд
  и сразу по изменению.
- LWT (`garage/heat/state` = `OFF`, retain) и пары `.../set`+`.../state` для
  каждой управляемой величины.

Подробное описание логики — в [`CLAUDE.md`](CLAUDE.md), §3.

## Логика регулирования: нюансы поведения

Формальное описание — в `CLAUDE.md` §3. Здесь — то, что легко понять неверно при первом
знакомстве с кодом: где реакция плавная, где жёсткий порог, а где — намеренная задержка.

### Главный контур пушки: не просто гистерезис по TARGET

Пуск (`TARGET < target − 0.5`) и останов (`TARGET > target + 0.5`) — это необходимое, но не
достаточное условие. Пуск (`OFF → FAN_STARTING`) дополнительно требует: MQTT-флаг
`fanHeater=ON`, валидные показания **и** TARGET, **и** BLOWN_AIR, и отсутствие активной аварии
радиатора (`radiatorAlarmState == NORMAL`, любой — не только критической, см. ниже). Без
проверки аварии радиатора автомат нагрева и радиаторная эскалация «спорили» бы за состояние
вентилятора пушки на каждой итерации `loop()` — авария принудительно гасит вентилятор и
сбрасывает автомат в `OFF`, а автомат тут же снова его включает, отсюда дребезг D5.

Пуск также не одновременный: сначала включается только вентилятор (D5), и лишь на следующем
шаге автомата — элемент (D6). Это не оптимизация, а следствие интерлока §3.4: элемент не может
включиться, пока программно не подтверждено, что вентилятор уже работает.

### BLOWN_AIR — плавная адаптация, не аварийный лимит

20°C (`target + sensorTempDiff`) — это не порог, при превышении которого что-то отключается.
Пока TARGET требует нагрева, элемент не горит постоянно, а циклится ON/OFF (duty-цикл); BLOWN_AIR
только подстраивает **соотношение** длительностей ON/OFF, не сам факт нагрева:

| BLOWN_AIR | Реакция |
|---|---|
| ≥ 20°C | поток перегрет → короче ON-фаза (не короче ¼ базового периода), длиннее OFF-фаза |
| ≤ 15°C | поток холодный → длиннее ON-фаза, короче OFF-фаза |
| 15–20°C | нейтральная зона, ON-фаза не меняется |

Это пропорциональная подстройка каждые `DUTY_ADAPT_CHECK_PERIOD_MS` (10с), а не выключатель.
Если BLOWN_AIR совсем не отвечает — это другое дело: тогда срабатывает `mustStop`, и элемент
гасится полностью, а не просто перестаёт адаптироваться.

### Останов пушки — по таймеру, не «до остывания»

При превышении TARGET+0.5 элемент гасится немедленно, но последующая продувка вентилятором —
**фиксированное время** (`fanCoolerDelay`, 10–20с по таблице §3.1 в зависимости от уличной
температуры), а не ожидание, пока что-то реально остынет по показаниям датчика. Температурный
порог остывания элемента нигде не измеряется — просто таймер.

### Радиатор (D4) + вентилятор охлаждения (D8): мгновенное включение, задержка на выключение

В отличие от всех остальных термоконтуров прошивки (TARGET ±0.5°C, AirTemp ±1°C), включение D8
по 30°C — это прямое сравнение без мёртвой зоны и без задержки: как только показание достигает
порога, `fanOn` в тот же момент становится `true`. Выключение, наоборот, **отложенное**: не
раньше чем через `RADIATOR_FAN_OFF_DELAY_MS` (60с) **непрерывно** ниже 30°C — защита от дребезга
D8 при температуре, колеблющейся около порога. Если за время ожидания температура снова
поднимается до 30°C — таймер выключения отменяется, вентилятор продолжает работать. Датчик
радиатора читается раз в 10с (`RADIATOR_SENSOR_POLL_PERIOD_MS`), так что и включение, и отсчёт
задержки выключения «квантованы» по 10-секундным интервалам, а не непрерывны.

Важный нюанс: **D8 не входит в принудительное отключение при аварии.** Когда объявляется
`FAN_FAULT`/`OVERTEMP`, гасятся элемент (D6), calorifer (D7) и вентилятор пушки (D5) — но не D8:
он продолжает управляться только своим порогом 30°C (а при отказе самого датчика D4 —
принудительно включается, а не выключается: во время перегрева выключать охлаждение было бы
контрпродуктивно).

### FAN_FAULT и OVERTEMP — одинаково жёстко, разница только в условии срабатывания

`FAN_FAULT` (33°C + минимум 60с работы вентилятора) и `OVERTEMP` (безусловно 35°C) приводят к
**одинаковому** результату — полному немедленному отключению элемента, calorifer'а и вентилятора
пушки одним вызовом, без какой-либо градации. Разница между ними только в том, при каком условии
ставится диагноз, а не в том, что происходит после.

Это отличается от обычного останова по TARGET ещё в одном: там вентилятор пушки (D5) продолжает
работать `fanCoolerDelay` после элемента, а при аварии радиатора D5 гасится **тоже немедленно**,
без всякой продувки — продлевать работу чего-либо на перегретом радиаторе не вариант.

Сброс аварии — тоже порог, не плавность: ниже 30°C (`RADIATOR_FAN_ON_TEMP` — тот же порог, что
включает вентилятор охлаждения) авария снимается автоматически по валидному опросу датчика, без
MQTT-подтверждения. Отдельного более низкого порога сброса нет намеренно — иначе сброс зависел бы
от сезона.

### Calorifer (D7) — независимый контур, но с той же радиаторной блокировкой

Гистерезис ±1°C вокруг `targetAirTemp` по датчику уличного воздуха (D2) — отдельный от пушки
контур, свой датчик, свой таргет. Но включение дополнительно требует: MQTT-флаг `calorifer=ON`,
валидные показания D2, и отсутствие **любой** активной аварии радиатора (`FAN_FAULT` блокирует
так же, как `OVERTEMP`, не только критический порог) — потому что обе SSR (пушки и calorifer'а)
физически сидят на одном радиаторе. К показанию D2 перед сравнением применяется фиксированная
калибровочная поправка −1°C.

### Активный реконнект WiFi/MQTT + watchdog как крайний случай

Watchdog — не первая линия обороны. Пока связи нет, прошивка сама пытается её восстановить:
`WiFi.reconnect()`/`WiFi.begin()` каждые `WIFI_RECONNECT_PERIOD_MS` (15 с — ядро ESP8266 Arduino
само не переподключается после разрыва, это подтверждено официальной документацией), и
`mqttClient.connect()` каждые `MQTT_RECONNECT_PERIOD_MS` (5 с), когда WiFi уже есть. 15 секунд для
WiFi — не наугад: WPA2-handshake и DHCP обычно занимают несколько секунд, а слишком частый повтор
`WiFi.begin()` посреди уже идущей попытки подключения мешает ей, а не ускоряет.

Только если оба этих механизма не справились, включается watchdog. 5 минут (`WATCHDOG_TIMEOUT_MS`)
— это осознанный grace-период, а не квантование по опросу (как у D8) и не фиксированная продувка
(как у D5). Пока WiFi **и** MQTT одновременно подключены, таймер простоя сбрасывается на каждой
итерации `loop()` — то есть фактически всегда держится на нуле, пока связь жива. Отсчёт стартует
только в момент потери любого из двух условий и должен пройти **непрерывно** — если связь на
секунду восстановится и снова пропадёт, отсчёт начинается заново с нуля, а не накапливается
суммарно за час. По истечении — единственное жёсткое действие без промежуточной эскалации: все
четыре силовых выхода гасятся безусловно, и следом `ESP.restart()`.

## Железо

| Назначение | Пин (Wemos D1 mini) | Компонент |
|---|---|---|
| Вентилятор тепловой пушки (триак) | D5 | MOC3066 (опторазвязка) → BTA16 |
| Нагревательный элемент тепловой пушки | D6 | SSR на медном радиаторе, через IRLML2402 |
| Вспомогательный нагреватель | D7 | SSR на медном радиаторе, через IRLML2402 |
| Вентилятор охлаждения радиатора SSR | D8 | через IRLML2402, порог включения 30°C |
| 3× DS18b20 на одном 1-Wire (BLOWN_AIR/TARGET/INFO) | D3 | роли по ROM-адресу, не по порядку |
| DS18b20 радиатора SSR | D4 | контроль температуры медного радиатора |
| DS18b20 уличного воздуха | D2 | одиночный датчик, 1-Wire |

Три датчика DS18b20 на шине D3 физически одинаковы — прошивка различает их не
по порядку подключения, а по зашитому в `include/config.h` 64-битному
ROM-коду. Для съёма реальных адресов есть отдельный инструмент —
см. [Инструменты](#инструменты).

Вентилятор тепловой пушки (D5) не имеет аппаратной обратной связи о вращении —
это готовый узел, управляемый только включением/выключением триака; см.
`CLAUDE.md` §0/№12 и §3.4.

## Архитектура прошивки

- `lib/control_logic/` — чистая, не зависящая от Arduino/ESP8266 логика
  (гистерезис, адаптация duty-цикла, эскалация радиатора, watchdog,
  валидация). Полностью покрыта unit-тестами (`env:native`, Unity).
- `src/main.cpp` — тонкая обвязка: чтение сенсоров (DallasTemperature,
  async-режим), запись GPIO, MQTT (PubSubClient), таймеры на `millis()`.
  Не блокирует `loop()` вызовами `delay()`. Все побочные эффекты и
  единственные точки записи в силовые выходы находятся здесь.
- `include/config.h` — некритичные константы/пороги/таймауты/топики.
- `include/secrets.h` — креды (WiFi SSID/pass, MQTT broker/user/pass), в репозиторий
  не коммитится (см. `include/secrets.h.example`).

Такое разделение позволяет тестировать всю регулирующую логику на native-платформе
без необходимости в реальном железе или эмуляторе ESP8266.

## MQTT

Устройство одно — ID в топиках не используется, корневой топик `garage/heat/`
используется напрямую. Без TLS, порт 1883 (локальная сеть).

Единого JSON-статуса нет — каждая величина публикуется в свой собственный
топик, простым текстом (не JSON). Схема: у управляемых величин — пара
`.../set` (команда, подписка устройства) и `.../state` (подтверждённое
устройством значение, publish с `retain`, обновляется сразу по приёму команды
и повторно раз в 60 секунд как heartbeat). У чисто телеметрийных величин
(сенсоры, физическое состояние выходов, диагностика) — только `.../state`.

`garage/heat/state` — `"ON"`/LWT `"OFF"` (retain) — доступность устройства
целиком, не путать с `.../state` топиками отдельных величин.

**Команды** (подписка, `.../set`):

| Топик | Назначение |
|---|---|
| `garage/heat/fanHeater/set` | вкл/выкл основной тепловой пушки |
| `garage/heat/targetSensorTemp/set` | целевая температура хранения (TARGET) |
| `garage/heat/sensorTempDiff/set` | порог адаптации duty-цикла (валидация 5..50) |
| `garage/heat/calorifer/set` | вкл/выкл вспомогательного нагревателя |
| `garage/heat/targetAirTemp/set` | целевая температура для вспомогательного нагревателя |
| `garage/heat/restart/set` | безопасное выключение всех силовых выходов + `ESP.restart()` |

**Телеметрия** (публикация, `.../state`) — см. таблицу в разделе
«Телеметрия» ниже.

Опциональный debug-топик (`garage/heat/debug`) — для лога, отдельно от
телеметрии. Сейчас никуда не подключён (события отладки идут только в
Serial, см. ниже).

## Управление

Всё управление и телеметрия — через MQTT, никакого веб-интерфейса или кнопок
на плате нет. Broker/user/pass — из `include/secrets.h`, порт 1883, без TLS.

### Команды

```bash
# основная тепловая пушка
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/fanHeater/set -m ON
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/fanHeater/set -m OFF

# целевая температура хранения (TARGET), дефолт 5.0
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/targetSensorTemp/set -m 5.0

# порог адаптации duty-цикла (валидация 5..50, вне диапазона — тихо отклоняется)
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/sensorTempDiff/set -m 15

# вспомогательный нагреватель
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/calorifer/set -m ON

# целевая температура для вспомогательного нагревателя, дефолт 10.0
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/targetAirTemp/set -m 10.0

# безопасный перезапуск: гасит все силовые выходы и делает ESP.restart()
mosquitto_pub -h <broker> -u <user> -P <pass> -t garage/heat/restart/set -m 1
```

Особенности:

- `fanHeater`/`calorifer` включаются строго значением `"ON"`; любое другое
  значение (в т.ч. пустое) трактуется как `OFF`.
- `sensorTempDiff` вне диапазона 5..50 молча игнорируется, старое значение
  сохраняется (при `DEBUG_LOG=1` в Serial будет видно `... rejected, out of
  range`) — и, соответственно, `.../state` не перепубликуется.
- `restart` срабатывает по любому payload — содержимое не проверяется.
- Все таргеты и флаги живут только в RAM: после перезагрузки (в т.ч. через
  `restart`) сбрасываются к дефолтам из `config.h`. Чтобы устройство не
  потеряло последнюю команду после `ESP.restart()`, отправляйте команды с
  флагом `retain` (см. `-r` у `mosquitto_pub` или `retain: true` в
  HA-интеграции) — устройство переподпишется и получит от брокера последнее
  сохранённое значение заново.
- После приёма и применения команды устройство сразу публикует
  подтверждение в соответствующий `.../state` (retain) — не нужно ждать
  минутного heartbeat, чтобы увидеть, что команда принята.

### Телеметрия

```bash
mosquitto_sub -h <broker> -u <user> -P <pass> -t 'garage/heat/+/state' -v
mosquitto_sub -h <broker> -u <user> -P <pass> -t 'garage/heat/state' -v
```

Каждая величина — свой топик `garage/heat/<имя>/state`, публикуется раз в
60 секунд (heartbeat) и сразу при изменении (для управляемых величин — по
приёму команды). Payload — простой текст, без JSON. Булевы величины —
буквально `"ON"`/`"OFF"`. Для датчика температуры отвалившийся сенсор даёт
payload `"None"` (специальное значение, которое HA MQTT sensor трактует как
`unknown`).

| Топик | Тип | Описание |
|---|---|---|
| `blownAirTemp/state` | float / `None` | датчик обдува пушки (в потоке горячего воздуха от элемента) — вход адаптации duty-цикла, CLAUDE.md §2/§3.1; сам по себе не участвует в таргете хранения |
| `targetTemp/state` | float / `None` | целевой датчик зоны хранения — по нему держится `targetSensorTemp` с гистерезисом `targetHysteresis`, §3.1 |
| `infoTemp/state` | float / `None` | информационный датчик — ни в каких расчётах не участвует, только для наблюдения, §2 |
| `outdoorTemp/state` | float / `None` | уличный воздух с калибровочной поправкой — вход термоконтура вспом. нагревателя и таблицы `fanCoolerDelay`, §3.1/§3.2 |
| `radiatorTemp/state` | float / `None` | радиатор SSR (общий для D6/D7) — вход эскалации перегрева `radiatorAlarm`, §3.3 |
| `radiatorAlarm/state` | `NORMAL`/`FAN_FAULT`/`OVERTEMP` | эскалация защиты радиатора, §3.3 |
| `fanOn/state` | `ON`/`OFF` | выход D5 — вентилятор тепловой пушки (триак) |
| `elementOn/state` | `ON`/`OFF` | выход D6 — нагревательный элемент пушки; не включается без `fanOn`, §3.4 |
| `auxOn/state` | `ON`/`OFF` | выход D7 — вспомогательный нагреватель |
| `radiatorFanOn/state` | `ON`/`OFF` | выход D8 — вентилятор охлаждения радиатора SSR |
| `fanHeater/state` | `ON`/`OFF` | текущее значение MQTT-флага `fanHeater/set` — разрешение работы контура пушки. Не то же самое, что `fanOn`/`elementOn`: флаг может быть `ON`, а выходы выключены (например, TARGET уже прогрет или идёт COOLDOWN) |
| `calorifer/state` | `ON`/`OFF` | то же самое для `calorifer/set` |
| `targetSensorTemp/state` | float | текущий таргет хранения, дефолт 5.0°C |
| `sensorTempDiff/state` | float | текущий порог адаптации duty-цикла, дефолт 15 |
| `targetAirTemp/state` | float | текущий таргет вспом. нагревателя, дефолт 10.0°C |
| `targetHysteresis/state` | float | гистерезис таргета хранения, константа ±0.5°C |
| `auxHysteresis/state` | float | гистерезис таргета вспом. нагревателя, константа ±1.0°C |
| `uptimeSeconds/state` | ulong | время с последней загрузки, секунды |
| `freeHeapBytes/state` | ulong | свободная куча ESP8266, байты — индикатор утечек памяти |
| `rssiDbm/state` | int | уровень сигнала WiFi, дБм (отрицательное число, ближе к 0 — сильнее) |
| `wifiConnected/state` | `ON`/`OFF` | подключение к WiFi (есть IP) |
| `mqttConnected/state` | `ON`/`OFF` | всегда `ON` в момент публикации (устройство не может опубликовать, будучи отключённым от брокера) |

`garage/heat/state` — `"ON"` (retain) пока прошивка жива; `"OFF"` — LWT,
брокер проставляет сам при обрыве TCP-сессии (устройство пропало из сети или
зависло).

## Отладочный вывод в Serial

Флаг `DEBUG_LOG` в `include/config.h` (по умолчанию `1`) включает вывод событий в
Serial (115200 baud) во время работы: переходы состояний автомата тепловой пушки,
переходы аварий радиатора, изменения каждого силового выхода (D5–D8),
подключение/отключение датчиков DS18b20, полученные и отклонённые MQTT-команды,
подключение к брокеру, публикация статуса, потеря/восстановление связи и
срабатывание watchdog. `DEBUG_LOG = 0` полностью убирает вывод из прошивки
(компилируется в no-op, не занимает Flash/время). Смотреть лог:

```bash
pio device monitor -b 115200
```

## Сборка и тестирование

Проект — [PlatformIO](https://platformio.org/). Два окружения:

```ini
[env:esp8266]   # прошивка, espressif8266 / d1_mini / arduino
[env:native]    # unit-тесты lib/control_logic на хосте (Unity)
```

Запуск unit-тестов чистой логики (не требует железа):

```bash
pio test -e native
```

Сборка прошивки:

```bash
pio run -e esp8266
```

Заливка (только по UART, OTA не реализовано):

```bash
pio run -e esp8266 -t upload
pio device monitor -b 115200
```

### Первый запуск

1. Скопировать пример и заполнить реальными кредами:

   ```bash
   cp include/secrets.h.example include/secrets.h
   ```

   `include/secrets.h` в `.gitignore` — не коммитится.

2. Снять реальные ROM-адреса трёх DS18b20 на шине D3 (см. ниже) и убедиться,
   что они совпадают со значениями `ROM_BLOWN_AIR`/`ROM_TARGET`/`ROM_INFO` в
   `include/config.h` (для нового комплекта датчиков — переснять и обновить).
3. `pio run -e esp8266 -t upload`.

## Home Assistant

Готовый YAML-файл для статической (не через MQTT Discovery) интеграции —
[`homeassistant/garage_heat.yaml`](homeassistant/garage_heat.yaml). 11 сенсоров,
7 бинарных сенсоров (состояния выходов + диагностика), два переключателя
(`fanHeater`, `calorifer`), три числовых таргета (`targetSensorTemp`,
`sensorTempDiff`, `targetAirTemp`) и кнопка перезапуска — сгруппированы в
одно устройство HA, по одному топику `.../state` на сущность (см. «MQTT» и
«Телеметрия» выше — единого JSON-статуса больше нет).

Файл сам начинается с top-level ключа `mqtt:` (обязательно — иначе HA
провалидирует сущности по устаревшей `platform:`-схеме и отклонит конфиг с
ошибкой `required key 'platform' not provided`) и подключается через
`packages` — это безопасно, даже если ключ `mqtt:` уже используется где-то
ещё в вашей конфигурации:

```yaml
homeassistant:
  packages:
    garage_heat: !include homeassistant/garage_heat.yaml
```

Переключатели и числа читают реальное состояние из своих `.../state`
топиков (`fanHeater/state`, `calorifer/state` и т.д.), а не работают в
optimistic-режиме. Payload — простой текст (`"ON"`/`"OFF"` для булевых), не
JSON, поэтому `value_template` почти нигде не нужен — единственное
исключение — производный бинарный сенсор «Авария радиатора (проблема)»,
читающий `radiatorAlarm/state` и сравнивающий его с `"NORMAL"`.

## Вне рамок проекта

- OTA/веб-обновление — прошивка заливается только по UART.
- TLS для MQTT — не требуется, сеть локальная.
- Датчик влажности — не реализуем.
- Аппаратная обратная связь по вентилятору пушки (тахометр/датчик тока) — не
  реализуем, интерлок (§3.4 `CLAUDE.md`) только программный.

Подробности решений и обоснования — в `CLAUDE.md`, §0 и §6.
