# Garage Heat v2 — архитектура прошивки (C++/PlatformIO)

Статус: дизайн-документ, описывает **как** реализуется логика из `CLAUDE.md` (ТЗ). Не меняет ни
одного порога/константы/поведения из ТЗ — только код-архитектуру: структуру проекта, библиотеки,
модель выполнения, разбиение на функции, стратегию тестирования.

Реализация (`main.cpp`/`config.h`/`secrets.h`) по-прежнему не начата — пишем план по этому
документу через `writing-plans`, к коду не приступаем, пока план не согласован.

---

## 1. Структура проекта и тулчейн

PlatformIO, плата **Wemos D1 mini** (`board = d1_mini` в `platformio.ini`; если фактическая плата
другая — меняется одной строкой, на разбиение файлов не влияет), `framework = arduino`.

```
platformio.ini
src/main.cpp          — вся логика управления (единственный .cpp с логикой, по требованию ТЗ)
include/config.h      — некритичные константы/пороги/таймауты/топики/пины/ROM-адреса
include/secrets.h     — WiFi/MQTT креды, в .gitignore, не коммитится
include/secrets.h.example — шаблон secrets.h с пустыми полями, коммитится
test/                 — нативные (host) unit-тесты чистых функций-решателей
.gitignore            — secrets.h, .pio/
```

## 2. Библиотеки

| Библиотека | Назначение |
|---|---|
| `PubSubClient` | MQTT-клиент |
| `ArduinoJson` (v7) | сборка status-JSON и парсинг входящих команд, где нужно |
| `OneWire` + `DallasTemperature` | DS18b20, неблокирующий async-режим конвертации (`setWaitForConversion(false)`) |
| `ESP8266WiFi` (встроенная) | WiFi |

Выбор сделан в пользу самого распространённого и предсказуемого стека для ESP8266 + PlatformIO;
`DallasTemperature` в async-режиме снимает проблему блокирующего 750мс-ожидания конвертации,
которая в старой прошивке решалась через колбэк таймера.

## 3. Модель выполнения

Один `loop()` без RTOS/шедулер-библиотек. Каждый периодический процесс — независимый
millis()-таймер по шаблону:

```cpp
unsigned long lastX = 0;
const unsigned long PERIOD_X = ...;
if (millis() - lastX >= PERIOD_X) {
    lastX = millis();
    // ...
}
```

Независимые тикеры в `loop()`:

| Процесс | Период |
|---|---|
| Быстрый опрос датчиков (D3: BLOWN_AIR/TARGET/INFO, D2: улица) | 5 с |
| Опрос датчика радиатора (D4) | 60 с (по ТЗ §3.3/§5) |
| Публикация JSON-статуса | 60 с (по ТЗ §4/§5) |
| Проверка watchdog связи (WiFi+MQTT) | каждую итерацию `loop()`, дешёвая проверка таймстампа |
| `mqttClient.loop()` (обработка входящих, keepalive) | каждую итерацию |
| Неблокирующая попытка реконнекта MQTT | раз в 5 с, только если клиент отключён |

`loop()` не должен блокироваться (`delay()` не используется). DS18b20 читается в async-режиме:
issue-конвертация → неблокирующее ожидание по таймеру → чтение.

## 4. Автоматы состояний

**HeaterState** (контур тепловой пушки, D5+D6, ТЗ §3.1 и §3.4):

```cpp
enum class HeaterState { OFF, FAN_STARTING, ELEMENT_DUTY_ON, ELEMENT_DUTY_OFF, COOLDOWN };
```

Переходы: `OFF → FAN_STARTING` (TARGET < target-0.5) → `ELEMENT_DUTY_ON`/`ELEMENT_DUTY_OFF`
(циклирование по `loadOnLimit`/`loadOffLimit`) → `COOLDOWN` (TARGET > target+0.5 или отказ
TARGET/BLOWN_AIR — элемент гасится сразу, вентилятор ещё `fanCoolerDelay`) → `OFF`.

Интерлок §3.4 обеспечивается структурно самой формой автомата: состояние `ELEMENT_DUTY_ON`
(единственное, где элемент включён) достижимо только через `FAN_STARTING`, в котором вентилятор
уже включён. Это не отменяет дополнительную проверку в `setElement()` (см. §7) — она остаётся
последним рубежом на случай ошибки в самом автомате.

**AuxHeaterState** (D7, ТЗ §3.2):

```cpp
enum class AuxHeaterState { OFF, ON };
```

Простой двухпозиционный, гейтится MQTT-флагом `calorifer` и радиаторной аварией (§3.3): при
активной аварии `AuxHeaterState` принудительно `OFF` независимо от термоконтура по уличному
воздуху.

**RadiatorAlarmState** (ТЗ §3.3):

```cpp
enum class RadiatorAlarmState { NORMAL, FAN_FAULT, OVERTEMP };
```

`FAN_FAULT` и `OVERTEMP` дают идентичное действие (безусловное отключение D5/D6/D7), различаются
только для публикации причины в статусе/логе. Сброс в `NORMAL` — автоматический при остывании
радиатора ниже 15°C (ТЗ §3.3), из любого из аварийных состояний.

**Connectivity watchdog** (ТЗ §3.5): не отдельный enum, а таймстамп
`unsigned long lastFullyConnectedMillis`, обновляемый при одновременном WiFi+MQTT connected;
`ESP.restart()` при превышении `WATCHDOG_TIMEOUT` с последнего обновления.

## 5. Подсистема датчиков

Три физически независимые 1-Wire шины (не одна общая):

- **D3** — 3× DS18b20, роли BLOWN_AIR/TARGET/INFO по ROM-адресу (ТЗ §2)
- **D2** — 1× DS18b20, уличный воздух
- **D4** — 1× DS18b20, радиатор SSR

Каждая шина — свой экземпляр `OneWire`/`DallasTemperature`. Быстрый цикл (5с) инициирует
конвертацию на D3 и D2 параллельно (async, не блокирует), радиатор (D4) — отдельный цикл 60с,
независимый от быстрого.

### Обработка отказа датчика

Общий принцип: **отказ датчика → безопасное отключение того, что от него зависит**, с флагом в
статусе и автоматическим повторным опросом на следующем цикле (без ручного вмешательства).
Применительно к конкретным датчикам:

| Датчик | При отказе (`DEVICE_DISCONNECTED_C` и т.п.) |
|---|---|
| TARGET | Контур пушки полностью OFF (эквивалент «TARGET > target+0.5»), флаг аварии в статусе. Повтор попытки чтения каждый быстрый цикл (5с). |
| BLOWN_AIR | Штатное выключение контура пушки по обычной последовательности §3.1 (элемент сразу, вентилятор ещё `fanCoolerDelay`) — трактуется как штатный уход в OFF, не авария, т.к. защищает сам элемент от потери контроля перегрева потока. |
| INFO | Не участвует в управлении — просто `null`/NaN в JSON-статусе. |
| D2 (улица) | Форс-выключение вспомогательного нагревателя (уже в ТЗ §3.2), `fanCoolerDelay` сбрасывается на дефолт 20000мс (уже в ТЗ). |
| D4 (радиатор) | Трактуется как критический перегрев — безусловное отключение D5/D6/D7, как порог 35°C (ТЗ §3.3 п.2). Автосброс требует восстановления валидного чтения ниже 15°C. |

## 6. MQTT

- `PubSubClient`, неблокирующий реконнект: попытка раз в 5с, пока не подключён; никогда не
  блокирует `loop()`.
- **Без очередей.** Статус-JSON — не история событий, а перезаписываемый снимок состояния:
  публикуется раз в 60с, если клиент не подключён — публикация просто пропускается (следующая
  попытка через 60с). Debug-топик (опциональный, ТЗ §4) — та же логика прямой публикации без
  буферизации, но публикуется только по значимым событиям (переход состояния автомата, авария,
  подключение/отключение), не каждый цикл `loop()` — чтобы не заспамить топик.
- Входящие команды обрабатываются синхронно внутри колбэка `PubSubClient` — присвоение переменной
  + немедленное применение к соответствующей подсистеме. Очередь на приём не нужна: обработка
  одного сообщения — O(1), колбэк успевает отработать до следующего сообщения.
- Retained-эхо командных топиков при подписке после реконнекта/ребута — ожидаемое штатное
  поведение (устройство фактически стартует с последнего known-state, если он есть на брокере;
  если retained-сообщений нет — стартует с дефолтов `config.h`). Не боремся с этим в коде.
- LWT на `garage/heat/state` = `"OFF"`, retain — как в ТЗ.

## 7. Структурная защита критичного пути (интерлок §3.4, радиатор §3.3)

Единственные точки записи в GPIO нагревательного элемента (D6) и вентилятора пушки (D5):

```cpp
void setFan(bool on);
void setElement(bool on);
```

`digitalWrite` на D5/D6 нигде больше не вызывается — это структурно исключает обход интерлока.
`setElement(true)` проверяет текущее логическое состояние выхода вентилятора и отказывает (с
логированием), если вентилятор не включён — согласно ТЗ §3.4, это защита от ошибок
логики/порядка вызовов в самом коде, не от физического отказа вентилятора/симистора/оптрона.

Радиаторная эскалация (ТЗ §3.3) реализована как отдельная функция, вызываемая на каждом
60-секундном цикле опроса D4, которая может форсировать `setFan(false)`/`setElement(false)`/
выключение D7 **независимо** от текущего `HeaterState`/`AuxHeaterState` — эскалация не завязана на
автоматы контуров, чтобы авария гарантированно перебивала любое их состояние.

Безопасная инициализация D5–D8 = LOW — первые строки `setup()`, до WiFi/MQTT/сенсоров (уже в ТЗ).

## 8. Тестирование без реального железа

ROM-адреса трёх DS18b20 на D3 ещё не сняты — реальное железо для интеграционных тестов
недоступно. Логика проверяется через **чистые функции-решатели** + PlatformIO native unit-тесты
(`pio test -e native`, без ESP8266-таргета):

- выбор `loadOnLimit`/`loadOffLimit` по температуре BLOWN_AIR (ТЗ §3.1)
- гистерезис TARGET (вкл/выкл контура пушки)
- выбор `fanCoolerDelay` по уличной температуре (таблица ТЗ §3.1)
- термоконтур вспомогательного нагревателя по уличному воздуху (ТЗ §3.2)
- эскалация радиатора: (температура, `fanOnSince`, текущее время) → `RadiatorAlarmState` (ТЗ
  §3.3, включая эвристику отказа вентилятора 33°C/60с)

Эти функции принимают только примитивы (float, `unsigned long` таймстампы/дельты) и возвращают
решения/enum — без прямых обращений к пинам, `millis()` изнутри или объектам библиотек. Вызываются
идентично из `main.cpp` и из нативных тестов. Полная интеграционная проверка (реальные датчики,
реальные SSR) возможна только на железе после снятия ROM-адресов — вне рамок этого документа.

## 9. config.h / secrets.h

`config.h` — все константы из ТЗ §5 (пороги, гистерезисы, таймауты, таблица `fanCoolerDelay`),
назначения пинов, топики MQTT, плейсхолдеры под ROM-адреса трёх DS18b20 (заполняются после
аппаратного шага снятия адресов — ТЗ §2/§7).

`secrets.h` — `WIFI_SSID`, `WIFI_PASS`, `MQTT_BROKER`, `MQTT_USER`, `MQTT_PASS`; в `.gitignore`.
`secrets.h.example` с пустыми/плейсхолдерными значениями — коммитится как шаблон для настройки.
