# Garage Heat v2 — ТЗ на прошивку

Статус: **ТЗ закрыто, реализация в процессе** (см. `progress.md` за статусом по задачам).

Прошивка — проект на C++ (Arduino core / PlatformIO), ESP8266. Архитектура: **один `main.cpp`**
со всей логикой, **`config.h`** — некритичные константы/пороги/таймауты/топики, **`secrets.h`** —
креды (WiFi SSID/pass, MQTT broker/user/pass), не коммитится.

Устройство подогревает гараж/место зимнего хранения мотоцикла, удерживая целевую температуру.

## 1. Железо

| Назначение | Пин ESP (Dx) | Компонент |
|---|---|---|
| Вентилятор тепловой пушки (триак) | D5 | MOC3066 (опторазвязка) → BTA16 |
| Нагревательный элемент тепловой пушки | D6 | SSR на медном радиаторе, через IRLML2402 |
| Вспомогательный нагреватель (вторая нагрузка/розетка) | D7 | SSR на медном радиаторе, через IRLML2402 |
| Вентилятор охлаждения радиатора SSR | D8 | через IRLML2402, порог включения 30°C |
| 3× DS18b20 последовательно на одном 1-Wire | D3 | см. §2 — роли по функции |
| DS18b20 радиатора SSR | D4 | контроль температуры медного радиатора (обе SSR на нём) |
| DS18b20 уличного воздуха | D2 | одиночный датчик, 1-Wire |

Вентилятор тепловой пушки (D5) не имеет никакой физической обратной связи о вращении — это
готовый узел, управляемый только включением/выключением триака.

## 2. Роли трёх DS18b20 на общем проводе (D3)

Три физических датчика на одной шине энумерируются не по порядку подключения, а по
детерминированной сортировке ROM-адресов. ROM-коды сняты с реального железа сканером
`tools/scan_ds18b20` (2026-07-13) и прописаны в `config.h`:

| Роль | ROM-адрес |
|---|---|
| BLOWN_AIR | `{ 0x28, 0x59, 0x79, 0x45, 0x92, 0x0F, 0x02, 0x9A }` |
| TARGET | `{ 0x28, 0xAA, 0x92, 0x24, 0x18, 0x13, 0x02, 0xAD }` |
| INFO | `{ 0x28, 0xAA, 0xB8, 0x85, 0x18, 0x13, 0x02, 0xCE }` |

Роли по функции:

1. **Датчик обдува (BLOWN_AIR)** — установлен непосредственно в потоке горячего воздуха от
   нагревательного элемента тепловой пушки. Не участвует в основном таргете хранения, но служит
   обратной связью защиты от перегрева потока: подстраивает длительность ON/OFF в duty-цикле
   нагревательного элемента (см. §3.1).
2. **Целевой датчик (TARGET)** — установлен в зоне хранения. Держит целевую температуру (деф.
   5°C) с гистерезисом ±0.5°C — основной контур регулирования пушки.
3. **Информационный датчик (INFO)** — ни в каких расчётах не участвует, только публикуется в
   JSON-статусе.

## 3. Логика регулирования

### 3.1. Тепловая пушка (вентилятор-триак D5 + нагревательный элемент D6)

Главный контур поддержания температуры хранения (деф. **+5°C**, гистерезис **±0.5°C**) по
TARGET-датчику:

- TARGET > target+0.5 и пушка греет → выключить весь контур (элемент сразу, вентилятор — с
  задержкой, см. ниже).
- TARGET < target-0.5 и весь контур выключен (и вентилятор, и элемент) → включить.

Пуск (переход OFF → FAN_STARTING) дополнительно требует **валидного BLOWN_AIR** и
**отсутствия активной аварии радиатора** (`radiatorAlarmState == NORMAL`, см. §3.3) — тот же
критерий, что уже применяется как `mustStop` во время работы (ниже), только проверенный и на
старте тоже. Без проверки аварии радиатора автомат нагрева и радиаторная эскалация «спорили» бы
за состояние на каждой итерации главного цикла: авария принудительно гасит вентилятор и сбрасывает
автомат в OFF, а автомат тут же снова его включает — в результате вентилятор (D5) дребезжал бы
вкл/выкл на каждой итерации, пока авария активна.

**Включение**: сперва вентилятор (D5), затем — только после вентилятора — элемент (D6). Внутри
периода нагрева элемент циклится ON/OFF (duty-цикл) двумя длительностями `loadOnLimit`/
`loadOffLimit`, которые адаптируются по BLOWN_AIR-датчику:

- BLOWN_AIR ≥ (target + tempDiff, деф. tempDiff=15 → 20°C) → уменьшить `loadOnLimit` (не ниже
  `baseDelay/4`), `loadOffLimit = baseDelay/1.5` — меньше грева, больше продувки.
- BLOWN_AIR ≤ (target + tempDiff − 5 → 15°C) → увеличить `loadOnLimit`, `loadOffLimit = baseDelay/2.5`.
- иначе — нейтральная зона, `loadOffLimit = baseDelay/2`.

Важно: 20°C здесь — не аварийный лимит, а вход пропорциональной адаптации соотношения ON/OFF;
превышение само по себе ничего не отключает, элемент продолжает циклиться, просто с другим
соотношением длительностей. Полное отключение элемента по BLOWN_AIR происходит только если датчик
перестал отвечать (см. `mustStop` выше) — это отдельный, жёсткий случай, а не продолжение той же
адаптации.

**Выключение**: элемент (D6) гасится немедленно; вентилятор (D5) продолжает работать ещё
`fanCoolerDelay` мс после выключения элемента (продувка/дожиг остаточного тепла), затем гасится —
тепловая защита самого элемента. Это фиксированный по времени таймер, а не ожидание остывания по
показаниям какого-либо датчика.

`fanCoolerDelay` (базовый период duty-цикла и время дожига вентилятора) зависит от уличной
температуры (датчик D2):

| Уличная темп. | `fanCoolerDelay`, мс |
|---|---|
| > 0°C | 20000 |
| 0…-5°C | 17500 |
| -5…-10°C | 15000 |
| -10…-15°C | 12500 |
| < -15°C | 10000 |

### 3.2. Вспомогательный нагреватель (D7)

Управляется MQTT-флагом вкл/выкл (деф. `OFF`). Активен только если включён **и** нет активной
аварии радиатора (`radiatorAlarmState == NORMAL` — блокирует и промежуточный порог отказа
вентилятора FAN_FAULT, и критический OVERTEMP, §3.3, не только критический). Термоконтур по
уличному воздуху (датчик D2), таргет
`targetAirTemp` (деф. **10°C**), гистерезис **±1°C**:

- airTemp > target+1 → выключить.
- airTemp < target-1 → включить.

Показание airTemp с фиксированной калибровочной поправкой (−1°C). Если датчик D2 не отвечает —
форс выключение, `fanCoolerDelay` сбрасывается к дефолту 20000 мс.

### 3.3. Радиатор SSR (датчик D4) + вентилятор охлаждения (D8) — эскалация защиты от перегрева

Многоуровневая логика:

1. **≥ 30°C** (порог) — вентилятор охлаждения (D8) включён, иначе выключен. Порог без гистерезиса
   (в отличие от остальных термоконтуров прошивки) — включение/выключение по прямому сравнению с
   30°C на каждом опросе датчика (раз в минуту, см. ниже), без мёртвой зоны и без задержки на
   угасание. D8 не входит в принудительное отключение по пункту 2 ниже — во время аварии он
   продолжает управляться только этим порогом (а при отказе самого датчика D4 — принудительно
   включается, а не выключается: во время неопределённости с температурой отключать охлаждение
   было бы контрпродуктивно).
2. **≥ 35°C** (критический) — безусловно и одновременно гасятся ОБЕ нагрузки на радиаторе
   (элемент пушки D6 и вспомогательный нагреватель D7) и вентилятор-триак пушки (D5), причём D5 —
   тоже немедленно, без обычной продувки `fanCoolerDelay` (§3.1) — на аварийном радиаторе
   продлевать работу чего-либо не оправдано. Публикуется аварийное сообщение/флаг в статусе.
   Устанавливается латч аварии.

   **Отказ самого датчика радиатора (D4) — с debounce.** Один одиночный сбойный/отключённый
   опрос (типичная наводка на 1-Wire) не эскалирует мгновенно до `OVERTEMP` — до накопления
   `RADIATOR_SENSOR_FAULT_DEBOUNCE_COUNT` (деф. 3) подряд неудачных опросов вентилятор (D8)
   принудительно включается (безопасный дефолт при неопределённости), но авария и нагрузки не
   трогаются. Только когда счётчик подряд неудачных опросов достигает порога — эскалация до
   `OVERTEMP`, идентичная пункту выше. Любой валидный опрос сбрасывает счётчик в 0. Без этого
   debounce одна случайная наводка на шине могла заморозить `OVERTEMP` до следующего валидного
   опроса с температурой ниже порога автосброса.
3. **Детекция отказа вентилятора охлаждения без тахометра** — простой порог, без анализа тренда
   (вариант B): датчик радиатора (D4) опрашивается раз в **10 секунд**
   (`RADIATOR_SENSOR_POLL_PERIOD_MS`). Как только D8 включается (по порогу 30°C), фиксируется
   момент включения `fanOnSince`. Если при очередном опросе температура радиатора всё ещё
   **≥ 33°C** (промежуточный порог, между рабочим 30°C и критическим 35°C) И с момента
   `fanOnSince` прошло не менее **60 секунд** (`RADIATOR_FAN_MIN_RUNTIME_MS`, минимальное время
   работы вентилятора, чтобы не среагировать на тепловую инерцию сразу после включения) —
   считать вентилятор неисправным и эскалировать до полного отключения нагрузок немедленно, не
   дожидаясь порога 35°C. Период опроса (10 с) и минимальное время работы (60 с) — независимые
   константы, не привязаны друг к другу: проверка срабатывает на первом опросе, где условие
   `fanOnSince` уже выполнено (в пределах ~10 с после достижения 60-секундной отметки), а не
   обязательно на первом опросе после включения вентилятора. Результат идентичен пункту 2 — то же
   самое безусловное отключение теми же вызовами; разница только в условии, по которому ставится
   диагноз, никакой дополнительной градации между промежуточным и критическим порогом нет.

**Сброс аварии**: автоматический, при остывании радиатора ниже **30°C** (`RADIATOR_FAN_ON_TEMP`,
тот же порог, что включает вентилятор охлаждения) — тоже порог, не плавный переход, и требует
именно валидного опроса датчика (сбойный опрос сам по себе снять аварию не может, см. debounce
выше). Отдельного более низкого порога сброса нет намеренно — иначе сброс зависел бы от сезона
(зимой радиатор остывает до уличной температуры за минуты, летом — нет, и авария могла бы
годами не сбрасываться при повторных проверках логики вне отопительного сезона). Ручное
MQTT-подтверждение не требуется.

Явная безопасная инициализация всех силовых выходов (D5, D6, D7, D8 = LOW) в самом начале
загрузки, до какой-либо другой логики.

### 3.4. Критический интерлок: элемент пушки только при работающем вентиляторе

Отдельное жёсткое требование, отдельно от радиаторной эскалации в §3.3: элемент тепловой пушки
(D6) не должен включаться, если вентилятор пушки (D5, триак через MOC3066/BTA16) не включён.

Аппаратной обратной связи по вращению нет и не планируется — интерлок реализуется программно как
безусловное правило на уровне управления выходами: функция включения элемента сама проверяет
текущее состояние выхода вентилятора и отказывает (с логированием), если вентилятор не включён.
Это защищает от ошибок логики/порядка вызовов в коде, но не от физического отказа самого
вентилятора, симистора или оптрона — таких случаев аппаратно не различить без датчика тока/тахометра.

### 3.5. Watchdog связи (WiFi + MQTT)

Watchdog — это крайний случай, а не основной механизм восстановления связи. До него дело обычно
не доходит, потому что прошивка активно пытается сама переподключиться, пока связи нет:

- **WiFi**: ядро ESP8266 Arduino не переподключает станцию само по себе после разрыва (проверено
  по официальной документации — нужен явный `WiFi.reconnect()`/`WiFi.begin()`, автоматики нет).
  Пока `WiFi.status() != WL_CONNECTED`, прошивка каждые `WIFI_RECONNECT_PERIOD_MS` (15 с) заново
  вызывает тот же `connectWiFi()`, что и на старте (`WiFi.begin()` с теми же SSID/паролем). Период
  дольше, чем у MQTT-реконнекта, намеренно: WPA2-handshake и DHCP обычно занимают несколько
  секунд, и слишком частый повторный `WiFi.begin()` посреди уже идущей попытки подключения мешает
  ей завершиться, а не ускоряет реконнект.
- **MQTT**: пока клиент не подключён к брокеру (и WiFi уже есть), каждые
  `MQTT_RECONNECT_PERIOD_MS` (5 с) — попытка `mqttClient.connect()`.

Только если оба механизма не смогли восстановить связь и WiFi не в состоянии подключения (нет IP)
**или** MQTT-клиент не подключён к брокеру — непрерывно более **5 минут** — срабатывает
`ESP.restart()`. Реализация: таймер/счётчик, фиксирующий момент последнего перехода в состояние
«полностью подключено» (WiFi + MQTT), сбрасывается при восстановлении обоих условий. Пока оба
условия выполнены, отметка обновляется на каждой итерации `loop()` — то есть простой фактически
всегда на нуле, пока связь жива. Отсчёт — непрерывный, не накопительный: если связь на мгновение
восстановится и снова пропадёт, 5 минут начинают отсчитываться заново, а не суммируются за более
долгий период. Это единственная в прошивке намеренная задержка такой длины — введена специально,
чтобы кратковременные обрывы сети/брокера не приводили к перезагрузке, пока активные попытки
реконнекта ещё могут восстановить связь сами.

### 3.6. Отладочный вывод в Serial

Флаг `DEBUG_LOG` в `config.h` (по умолчанию `1`) включает вывод событий в Serial (115200
baud) для отладки: переходы состояний автомата пушки (`HeaterState`), переходы аварий радиатора
(`RadiatorAlarmState`), изменения состояния каждого силового выхода (D5–D8), подключение/
отключение каждого из пяти DS18b20, полученные и отклонённые MQTT-команды, подключение/
переподключение к MQTT-брокеру, публикация телеметрии, потеря/восстановление связи и
срабатывание watchdog. При `DEBUG_LOG = 0` весь вывод компилируется в no-op (макросы `DBG_*` в
`main.cpp`) — не занимает ни Flash, ни времени выполнения. Отказ интерлока §3.4 (попытка включить
элемент без вентилятора) логируется всегда, независимо от `DEBUG_LOG`, — это сигнал ошибки логики,
а не отладочная информация.

## 4. MQTT

- Без ID устройства — единственное устройство, корневой топик `garage/heat/` используется
  напрямую.
- LWT: `garage/heat/state` = `"OFF"`, retain — доступность устройства целиком (не путать с
  `.../state` отдельных величин ниже).
- Без TLS, порт 1883, локальная сеть — брокер/логин/пароль в `secrets.h`.
- **Схема топиков — одна величина = один топик, без единого JSON-статуса.** У каждой
  управляемой величины — пара топиков: `.../set` (команда, подписка устройства) и `.../state`
  (подтверждённое устройством значение, публикация с `retain`). У чисто телеметрийных величин
  (сенсоры, физическое состояние выходов, диагностика) — только `.../state`. Payload — простой
  текст, не JSON; булевы величины публикуются буквально как `"ON"`/`"OFF"`.
- Топики команд (подписка, `.../set`):
  - `garage/heat/fanHeater/set` (вкл/выкл основной пушки)
  - `garage/heat/targetSensorTemp/set`
  - `garage/heat/sensorTempDiff/set` (диапазон валидации 5..50)
  - `garage/heat/calorifer/set` (вкл/выкл вспомогательного нагревателя)
  - `garage/heat/targetAirTemp/set`
  - `garage/heat/restart/set` → безопасное выключение всех силовых выходов + `ESP.restart()`
    (действия без состояния, `.../state` не публикуется)
- Подтверждение команды (публикация, `.../state`) для каждой из пяти управляемых величин выше
  (кроме `restart`) публикуется сразу по приёму и применению команды — не нужно ждать
  периодической перепубликации, чтобы увидеть, что команда принята. `fanHeater/state` и
  `calorifer/state` — это именно значение MQTT-флага, не физический выход: флаг может быть
  `ON`, а выход выключен (например, TARGET уже прогрет или идёт COOLDOWN); нужны, чтобы внешние
  системы вроде Home Assistant могли отражать реальное состояние команды, а не последнее
  отправленное значение (optimistic).
- Телеметрия (публикация, только `.../state`, без команд): температуры всех сенсоров
  (BLOWN_AIR/TARGET/INFO, уличный воздух, радиатор), эскалация радиатора (`radiatorAlarm/state`
  — `NORMAL`/`FAN_FAULT`/`OVERTEMP`), физическое состояние всех выходов (D5–D8), гистерезисы
  (как константы для наблюдаемости), uptime, heap, RSSI, состояние WiFi/MQTT. Отвалившийся
  датчик температуры публикует payload `"None"` в свой `.../state` — специальное значение,
  которое MQTT-интеграция Home Assistant трактует как состояние `unknown`.
- **Периодичность**: все `.../state`-топики (и управляемые, и телеметрийные) перепубликуются
  целиком раз в **60 секунд** как heartbeat — независимо от изменения значения. Управляемые
  величины (пять `.../set`-величин выше) дополнительно публикуются немедленно по приёму команды
  (см. выше). Вся телеметрия принудительно перепубликовывается сразу после (пере)подключения к
  брокеру, чтобы подписчики не ждали до минуты после реконнекта, чтобы увидеть актуальные
  значения.
- **Критичные телеметрийные величины публикуются немедленно при изменении**, а не только раз в
  минуту — минутная задержка неприемлема для аварийных сигналов и состояния силовых выходов:
  `radiatorAlarm/state`, `radiatorTemp/state`, `fanOn/state`, `elementOn/state`, `auxOn/state`,
  `radiatorFanOn/state`, `targetTemp/state`, `blownAirTemp/state`, `outdoorTemp/state`,
  `wifiConnected/state`. Немедленная публикация — это дополнение к heartbeat, не замена: те же
  топики всё равно перепубликуются раз в минуту вместе со всеми остальными. `mqttConnected/state`
  в этот список не входит: публиковать её саму можно только пока клиент подключён, поэтому
  «мгновенное изменение» здесь совпадает с уже существующей немедленной `publishAllState()` сразу
  после реконнекта — отдельного отслеживания не требуется, факт разрыва связи покрывается LWT.
  **Вспомогательные величины** (`infoTemp/state`, `uptimeSeconds/state`, `freeHeapBytes/state`,
  `rssiDbm/state`) публикуются только по heartbeat раз в минуту, без немедленной публикации по
  изменению — не критичны для мониторинга/автоматизаций, не оправдывают лишний MQTT-трафик.
- Опциональный debug-топик для лога — отдельно от телеметрии.
- Очередь публикации/входящих команд — ограничить по размеру (не допускать неограниченного роста
  при долгом обрыве связи).

## 5. Параметры для `config.h`

- `TARGET_STORAGE_TEMP = 5.0`, гистерезис `0.5`
- `SENSOR_TEMP_DIFF = 15` (порог BLOWN_AIR = target+diff = 20°C, нижняя граница нейтральной зоны = target+diff−5 = 15°C)
- `TARGET_AUX_AIR_TEMP = 10.0`, гистерезис `1.0`
- `AIR_TEMP_CALIBRATION_OFFSET = -1.0`
- `RADIATOR_FAN_ON_TEMP = 30.0`
- `RADIATOR_CRITICAL_TEMP = 35.0`
- Порог сброса аварии — отдельной константы нет, используется `RADIATOR_FAN_ON_TEMP` (30.0)
- `FAN_COOLER_DELAY` — таблица по уличной температуре (10000–20000 мс, см. §3.1)
- `MQTT_STATE_PUBLISH_PERIOD = 60` c (heartbeat-период перепубликации всех `.../state`, см. §4)
- `WATCHDOG_TIMEOUT = 300` c (5 мин)
- ROM-адреса трёх DS18b20 на D3 (BLOWN_AIR/TARGET/INFO) — сняты с реального железа, см. §2
- `RADIATOR_FAN_FAULT_TEMP = 33.0` (промежуточный порог отказа вентилятора)
- `RADIATOR_FAN_MIN_RUNTIME = 60` c (минимальное время работы вентилятора перед проверкой)
- `RADIATOR_SENSOR_POLL_PERIOD = 10` c (период опроса датчика радиатора D4)
- `RADIATOR_SENSOR_FAULT_DEBOUNCE_COUNT = 3` (подряд неудачных опросов D4 до эскалации в OVERTEMP, см. §3.3)
- `DEBUG_LOG = 1` — отладочный вывод событий в Serial (см. §3.6), `0` — отключить

## 6. Явно вне рамок

- OTA/веб-обновление — не реализуем. Прошивка заливается только по UART.
- TLS для MQTT — не требуется, сеть локальная.
- Влажность/датчик влажности — не реализуем.
- Аппаратная обратная связь по вентилятору пушки (тахометр/датчик тока) — не реализуем, интерлок
  только программный (см. §3.4).
